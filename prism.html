📋 Problem Statement: Employee Management System

Mini Project 1: Employee Management System

1. Overview

Employee Management System is your first step in a two-part mini project series. In this project you will design and build a browser-based Employee Management System using only frontend technologies. All employee data is stored as a hardcoded JavaScript array of objects — no backend, no database, and no frontend frameworks are permitted.

What makes this project different from a typical beginner task is its emphasis on modular architecture. Rather than placing all logic in a single app.js file, the codebase is split across purpose-built service modules — mirroring how real-world frontend projects are structured. Each service has a single responsibility, making the code easier to read, test, and maintain.

In subsequent mini project, the static data layer will be replaced by a .NET Core Web API backed by SQL Server, and in-memory authentication will be upgraded to JWT-based login. The modular structure you build now makes that transition straightforward — swapping the data source requires changes to only one service file.

2. Problem Statement

Background

XYZ Solutions, a technology services company, currently manages employee records manually using spreadsheets. This approach causes data inconsistency, difficulty in tracking employee details, and a lack of centralised visibility for the HR team.

The company now requires a modern, responsive, browser-based Employee Management System that:

  • Allows an Admin to securely Sign Up and Login
  • Displays workforce insights through a real-time summary dashboard
  • Supports adding, editing, deleting, searching, filtering, and sorting employee records
  • Works entirely on the frontend without page reloads
  • Is built with a modular, service-oriented JavaScript architecture

Your Task

Design and develop a responsive Employee Management System using HTML5, CSS3, Bootstrap 5, JavaScript (ES6+), and jQuery 3.x. The application must include:

  • Admin Signup Page
  • Admin Login Page
  • Dashboard View — summary cards, department breakdown, recent employees
  • Employee List View — with search, filter, and sort
  • Complete CRUD Operations — Add, View, Edit, Delete

All employee data must be stored in data.js as a JavaScript array of objects. All UI updates must be handled dynamically using DOM manipulation — no full-page reloads. Application logic must be distributed across the designated service modules described in Section 7.

3. Admin Authentication Module

The application must begin with an authentication flow before any HR data is accessible. Authentication is simulated entirely in-memory. The authService.js module owns all authentication logic.

3.1 Admin Signup

The Signup page must allow an Admin to register with the following fields: Username, Password, Confirm Password.

Validation Rules (handled by validationService.js)

  • All three fields are required — empty submission must show field-level error messages
  • Password must be a minimum of 6 characters
  • Password and Confirm Password must match exactly
  • Username must be unique — duplicate signup must be rejected with a specific error message

On Successful Signup

  • authService.js stores the admin credentials as an object in memory (seeded via data.js)
  • User is redirected to the Login page
  • A Bootstrap Toast success notification is shown before redirect

3.2 Admin Login

The Login page must authenticate the Admin using: Username, Password.

Validation Rules

  • Both fields are required
  • Credentials must exactly match the data stored by authService.js
  • An error message must be shown for invalid credentials — do not reveal which field is incorrect

On Successful Login

  • authService.js sets an in-memory session flag to indicate the user is authenticated
  • User is redirected to the Dashboard (dashboard section in index.html)
  • Accessing the dashboard section without a valid session redirects back to the login view

4. Employee Data Model

All employee records are declared in data.js as a JavaScript array of objects. storageService.js provides the in-memory read/write interface over this array so that no other module accesses the raw data array directly.

FieldTypeDescription
idNumberUnique auto-incremented identifier (e.g., 1,2,3…)
firstNameStringEmployee's first name (e.g., 'Priya')
lastNameStringEmployee's last name (e.g., 'Menon')
emailStringUnique work email address (e.g., 'priya.menon@XYZ.com')
phoneString10-digit contact number (e.g., '9876543210')
departmentStringOne of: Engineering | Marketing | HR | Finance | Operations
designationStringJob title (e.g., 'Software Engineer', 'HR Executive')
salaryNumberAnnual salary as a positive number (e.g., 750000)
joinDateStringDate of joining in YYYY-MM-DD format (e.g., '2022-06-15')
statusString'Active' or 'Inactive'

Dataset Requirement: A minimum of 15 employees must be preloaded in data.js, distributed across all five departments with a mix of Active and Inactive statuses.

5. Core Features to Implement

5.1 Dashboard View

The Dashboard is the default landing page after login and is managed by dashboardService.js. It must display summary cards (Total, Active, Inactive, Departments), department-wise employee count, and a recent employees panel (last 5 added).

5.2 Employee List View

A dedicated section displaying all employees in a structured table with columns: ID, Full Name, Email, Department, Designation, Salary (formatted as ₹X,XX,XXX), Join Date, Status, Actions (View, Edit, Delete).

5.3 Search and Filtering

  • Real-time search by name/email · Department dropdown · Status toggle (All/Active/Inactive) · AND logic

5.4 Sorting

  • Sort by Name, Salary, Join Date in both directions, combined with active filters.

5.5 CRUD Operations

  • Create: 'Add Employee' button opens a Bootstrap Modal with form validation (validationService.js). On success, employeeService adds via storageService, UI refreshes, toast shown.
  • Read: View button opens read-only modal with employee details.
  • Update: Edit button pre-populates modal; changes saved via employeeService.
  • Delete: Confirmation modal; removal via employeeService.

6. Navigation & Layout

  • Fixed-top Bootstrap Navbar with app name, navigation links (Dashboard, Employees), 'Add Employee' button, and Logout.
  • Responsive: collapses on mobile.
  • Dashboard and Employee List sections coexist in index.html; toggled via jQuery.
